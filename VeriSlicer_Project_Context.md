## 1. Project Objective
VeriSlicer is an automated hardware debugging framework designed to identify the exact line number of single-line faults in complex Verilog RTL designs. Due to the massive token sizes of large codebases and VCD traces, VeriSlicer acts as an aggressive compiler and simulation filtering mechanism to pare down irrelevant data before passing the tightly constrained context to a locally hosted LLM (currently `qwen2.5-coder:14b`) for fault prediction.

## 2. Directory Structure

The project root is typically located near `Dataset/`.
```text
Dataset/
├── Designs/               # Original bug-free designs and their text `description.txt`
├── Testbenches/           # Cocotb Python testbenches responsible for simulation and driving stimuli
├── Error_designs/         # The core dataset of buggy designs
│   ├── error_design_1_1/  # Instance of a bug
│   │   ├── design_1.v           # The faulty Verilog code
│   │   ├── error_info.txt       # GROUND TRUTH: states the exact buggy line integer & root cause
│   │   └── error_timestamp.txt  # RUNTIME SYMPTOM: generated by cocotb (failing signal, expected vs actual value, failure timestamp)
│   └── ... 
└── Pyverilog_general/     # The primary working directory for the pipeline scripts
    ├── framework_local.py # Master orchestrator script 
    ├── mapper.py          # Pyverilog AST graph generator
    ├── latency_est.py     # BFS causal latency calculation
    ├── trimmer.py         # VCD tracer & trimmer
    ├── vcd_summarizer.py  # VCD to LLM-friendly Text converter
    ├── code_slicer.py     # Verilog pruning tool
    ├── llm_request.py     # Combines everything into the final LLM Prompt
    └── evaluate_llm_local.py # Local LLM interaction and scoring 
```

## 3. Execution Pipeline Step-by-Step
The central hub is `framework_local.py` which runs a massive loop over all `error_design_X_Y` folders identically:

1. **Simulation**: Generates a runtime `Makefile` invoking Icarus Verilog and Cocotb to run tests. Dumps the full signal activity as a `*.vcd` file.
2. **AST Mapping**: `mapper.py` parses the faulty Verilog via Pyverilog into a graph mapping signal dependencies.
3. **Latency Estimation**: The DFS algorithm in `latency_est.py` establishes the worst-case combinational depth from inputs to the *failing pin* designated by `error_timestamp.txt`.
4. **VCD Tracing**: `trimmer.py` ingests the vast output `.vcd` file and prunes it to a temporal window leading exactly up to the failure matching the computed latency. 
5. **VCD Summarization**: `vcd_summarizer.py` converts the trimmed VCD into dense ASCII Hex format for Token optimization. **Crucial insight**: It adaptively looks backward for an "active" delta window and *always* force-prints a snapshot of the exact failure tick so the LLM doesn't get blind-sided by signals that stabilized early.
6. **Code Slicing**: `code_slicer.py` strips the original Verilog to merely the assignments, block setups, and structures driving the failing graph. Importantly, it prepends the *original line numbers* from `design_X.v` to the left edge of each line.
7. **Prompt Construction**: `llm_request.py` stacks the Error Symptoms, the Text-VCD, the Module Spec, and the Sliced Code.
8. **Stateless LLM Evaluation**: `evaluate_llm_local.py` pings the Local Inferencing server (Ollama/vLLM) with a Temperature of 0.0 to predict the Top-5 most likely buggy lines.

## 4. Current State & Hard-won Optimizations

> [!IMPORTANT]
> The biggest challenge faced thus far was **LLM Integer Hallucination**. For a long time, the LLM inferred the theoretical fault perfectly and wrote pristine fixed Verilog strings, but utterly failed to provide the correct integer corresponding to the line in `buggy_lines` JSON output because it got confused by 10+ element code snippets.

**How we solved it:**
1. **Chain-of-Thought (CoT) Anchoring**: In `llm_request.py`, we heavily modified the JSON schema the prompt expects. The model must output `"buggy_code_snippet"` BEFORE it is allowed to output `"buggy_lines"`, forcing it to mentally extract the code string containing the line number *before* parsing the integer out.
2. **Deterministic Outputs**: `TEMPERATURE` is strictly set to `0.0`.
3. **Strict Line Requirements**: System prompts heavily highlight that it must copy the integer from the exact left-hand prefix in the Sliced Verilog output block. 

### Other Technical Fixes to be aware of:
- `Makefile` deprecation for `MODULE=` was mitigated by rewriting `framework_local.py`'s `create_makefile` to output `COCOTB_TEST_MODULES=`.
- `parse_llm_response()` in `evaluate_llm_local.py` writes broken JSON model responses to `raw_failed_response.txt` so we don't silently swallow Qwen timeouts/hallucinations for massive designs.

## 5. Metrics & Max Capabilities

- Current 14B model analytical ceiling is approximately **40-50% Exact-Line Localization Accuracy**. 
- It handles standard combinational routing errors beautifully. 
- It excels at interpreting numerical shifts (e.g. noticing actual is 2x expected and actively searching for `*2` or `<< 1` operators). 
- Extreme complexity components (e.g., SIMD CPU instruction set routing arrays) can generate prompts up to 34,000 characters (8k tokens) which the local GPU processes seamlessly but stretches 14B capabilities thinly.
